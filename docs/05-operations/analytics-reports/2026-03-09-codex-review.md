### 2-1. 494→13（-97.4%）の急減解釈
- 問題点:
  - 前週がスパイク週であることは事実だが、1週だけで「正常化」と結論づけるのは早い。
  - `スクリーンページビュー15 / 12ページ` は、自然流入としては薄すぎるため、計測欠損や抽出条件のズレも疑うべき。

  `Step A: 計測健全性確認（Realtime/DebugView/タグ配信）`

- 改善提案:
  - 週次レポートを `T+72h` の確定版運用に変更（暫定版と確定版を分離）。

### 2-3. `/glossary/ai-debate` 1,421秒の信頼性

- 改善提案:
  - ページ評価を `平均` から `中央値 + P75 + 最小母数条件` に変更（例: `PV>=20` で評価対象）。
  - 当該ページはイベント分解で検証（`session_start`, `user_engagement`, `scroll_depth_*`, CTAクリック有無, hostname, country, browser）。

## 5. Search Console問題への追加分析

### 「OAuth再認証が必要」診断の妥当性
- 妥当性は高いです。スコープ追加後に既存トークンが旧権限のまま残るケースは実際に起こります。
- ただし、「再認証だけが唯一の残作業」と断定するのは早く、以下の同時確認が必要です。

### 再認証以外に必要な確認項目
1. Apps Script が参照する Google Cloud Project で Search Console API が有効化されているか。  
2. 実行ユーザーに Search Console プロパティ権限があるか（Owner/Full User）。  
3. `siteUrl` 指定がプロパティ形式と一致しているか（`sc-domain:ai-reboot.io` vs `https://ai-reboot.io/`）。  
4. GASの実行コンテキスト（手動/トリガー）で同一アカウント認証になっているか。  
5. 再認証後に `ScriptApp.getOAuthToken()` でトークン更新を確認し、テストクエリで200系応答を取得できるか。  
6. Search Consoleデータ自体の反映遅延（通常1-3日）を前提に、復旧当日のゼロ値を障害扱いしない運用にすること。
